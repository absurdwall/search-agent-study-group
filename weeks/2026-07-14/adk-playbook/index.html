Set up Google ADK

A small, reproducible path from a fresh machine to a verified Google ADK environment. We are writing this guide while running every step ourselves, so it will improve as the study group learns more.

The setup path

  1. 1
    Clone the clean learner repository.
  2. 2
    Prepare its isolated local environment.
  3. 3
    Complete one optional live checkpoint.

Keep the setup isolated

Do not run a bare pip install google-adk inside a shared Anaconda, Spyder, or system Python environment. The lab creates its own .venv so ADK cannot upgrade packages used by other tools.

Use a network and device that permit the lab

A corporate VPN, proxy, firewall, or managed laptop may block GitHub, Python package downloads, or Google AI services. We recommend using a personal laptop for this learner project. If a network step fails, try a personal network or disconnect the VPN only when company policy permits it. Do not bypass employer security controls.

1

Download the learner repository

The learner repository contains the tested dependencies, setup notes, and a small setup-check notebook. Clone it, then enter the folder.

Choose one base folder

Set PROJECTS_DIR to the folder where you keep coding projects. Git will create search-agent-lab inside it automatically.

Example only · replace this path with your own
export PROJECTS_DIR="/your/base/directory"

Do not run that example unchanged. Replace /your/base/directory with a real folder on your computer before continuing.

Terminal · after setting PROJECTS_DIR
mkdir -p "$PROJECTS_DIR"
cd "$PROJECTS_DIR"
git clone https://github.com/absurdwall/search-agent-lab.git
cd search-agent-lab

Already cloned it? Run cd "$PROJECTS_DIR/search-agent-lab", then git pull before continuing.

2

Set up the isolated environment

Create a repository-local Python environment and install the exact dependency set that has already passed the lab’s offline check.

Terminal · from the repository root
cd "$PROJECTS_DIR/search-agent-lab"
python3.11 --version
python3.11 -m venv .venv
source .venv/bin/activate

python -m pip install --require-hashes -r requirements.lock
python -m pip check
adk --version

Prefer the official installation path?

Our requirements.lock keeps every learner on the same tested package versions. If you prefer to manage your own versions, keep the .venv active and run python -m pip install google-adk instead. Follow the official Python quickstart and installation guide. Do not run both installation commands in the same environment.

A successful setup ends with No broken requirements found. and an ADK version. Before running agent commands, verify that this terminal is using the lab environment:

Environment check
which python
which pip
which adk

All three paths should begin with the repository path and include .venv/bin/.

What the virtual environment does

python3.11 -m venv .venv creates an isolated Python environment inside this copy of the repository. Activating it makes this terminal use that Python and its packages. Each learner creates their own .venv; the folder is never uploaded to GitHub.

requirements.lock
Committed — the shared installation recipe
.venv/
Local only — generated on each learner’s computer
.env.example
Committed — a safe placeholder template
.env
Local only — contains the learner’s real API key

Coming back in a later terminal session

Creating and installing the environment is a one-time setup. On later runs, enter the repository and activate the existing environment:

Terminal · later sessions
export PROJECTS_DIR="/your/base/directory"
cd "$PROJECTS_DIR/search-agent-lab"
source .venv/bin/activate

When you are finished, run deactivate to leave the virtual environment.

3

Check the local environment

For now, we encourage every learner to run the repository locally on their own computer. This keeps the environment and branch workflow predictable. If there is strong demand, we can add a supported GitHub Codespaces path later.

Run the notebook in VS Code

Install the VS Code Python and Jupyter extensions, then open the search-agent-lab folder itself in VS Code. Open notebooks/00_setup_check.ipynb and:

  1. Click Select Kernel in the notebook’s upper-right corner.
  2. Choose Select Another Kernel… → Python Environments.
  3. Select the Python interpreter inside search-agent-lab/.venv.
  4. Replace GITHUB_USERNAME = "your-github-username" with your public GitHub login.
  5. Run all cells.

The locked setup already installs ipykernel, so no extra package installation or VS Code restart should normally be necessary. The first part checks the Python version, ADK installation, repository, checkpoint utilities, and credential detection. Without a key, it makes no model request and leaves the live checkpoint waiting.

If the .venv kernel does not appear

Activate the environment and register it once, then run Developer: Reload Window from the VS Code Command Palette:

Terminal · fallback only
cd "$PROJECTS_DIR/search-agent-lab"
source .venv/bin/activate
python -m ipykernel install --user --name "search-agent-lab" --display-name "Search Agent Lab"
Expected result Local environment check: PASS
Expected without GOOGLE_API_KEY
credential: not configured
Local environment check: PASS
Live agent checkpoint is waiting for GOOGLE_API_KEY; no model request was made.

The committed notebook stays output-free. Clear saved cell output before committing changes.

4

Create a Gemini API key

Open Google AI Studio — API Keys, sign in, and create a key for your own use.

Choose a project and name the key

Select Default Gemini Project, or another Free-tier project that you personally own. A separate project is optional; it is useful only when you deliberately want separate billing, quota, permissions, or collaborators. Use a recognizable key name such as search-agent-lab.

What is free?

If the selected project shows Free under Billing Tier, the initial lab can use Gemini 3.5 Flash input and output at no charge within the model’s Free-tier limits. A Google AI Pro subscription is not required. Do not enable billing for this exercise. On the Free tier, Google may use the content you send and receive to improve its products. Do not use Walmart information, personal data, confidential material, or other sensitive content; use only public information or invented examples. Check current pricing.

  • Use your own key; do not share one key across the group.
  • Store it only in the repository’s local .env file. Before pasting the key, run git check-ignore .env; the expected output is .env, confirming that Git will not upload it.
  • Do not paste it into chat, a recording, notebook output, or Git.

Recording this step?

Pause or hide the recording before the key appears on screen. If a key appears in a screenshot, chat, terminal transcript, or video, delete it immediately and create a replacement. Never reuse the exposed key.

Store the key in the local repository

A key may appear as a long string beginning with something like AQ.…; the exact format can change. Treat the entire value as a password and never show it in screenshots, recordings, or messages. Create the private local file before pasting the key:

Terminal
cd "$PROJECTS_DIR/search-agent-lab"
cp .env.example .env
git check-ignore .env .venv

The last command should list both .env and .venv, confirming that Git ignores them. Open .env in a text editor, then paste the replacement key only into that local file.

Do not put the real key in .env.example. That file is a public, tracked template; .env is the private, ignored copy.

Inside .env · never commit this file
GOOGLE_API_KEY="your-own-key-goes-here"
5

Complete the live agent checkpoint

Return to notebooks/00_setup_check.ipynb with the same .venv kernel and run all cells again. When the notebook detects GOOGLE_API_KEY, it automatically makes one small Gemini-backed ADK invocation—there is no live-mode switch to change. Raw ADK events pass through an allowlisted adapter before anything is displayed or assessed.

Expected successful timeline
Redacted runtime timeline
user input    | Use lookup_lab_status for the public topic google-adk, then give one short setup status.
agent         | setup_agent
tool call     | {'tool': 'lookup_lab_status', 'arguments': {'topic': 'google-adk'}}
tool result   | {'tool': 'lookup_lab_status', 'result': {'status': 'ready', 'topic': 'google-adk', 'summary': 'The deterministic local tool completed.'}}
final answer  | Final response received (content omitted by setup check).

The checkpoint unlocks only after the real run contains the expected tool call, exact allowlisted tool result, and a non-thought final-answer event. The notebook then combines that validated evidence with your GitHub username to produce a deterministic weekly codename.

Maintainer example · your codename will be different
🎉 The agent found its first tool!
Week 1 agent codename: 🟣 Violet Dolphin — Agent Trailblazer
Optional public Issue Form: https://github.com/absurdwall/search-agent-lab/issues/new?...

🟣 Violet Dolphin — Agent Trailblazer is the maintainer’s codename. Your result is generated from your own GitHub username and the validated checkpoint evidence, so expect a different color, animal, or agent title.

6

Submit the checkpoint (optional)

Submission is separate from setup. Use it only if you want a public, lightweight confirmation that your Week 1 checkpoint completed.

  1. Open the prefilled Issue Form URL printed by the notebook.
  2. Confirm the checkpoint ID, checkpoint phrase, and codename are already filled in.
  3. Check I completed the optional live checkpoint and am sharing only the generated public codename.
  4. Select Create. Do not add notebook output, paths, prompts, or credentials.
  5. Wait briefly while GitHub Actions validates the actual Issue author.

What happens after submission?

A valid submission receives checkpoint, passed, and week-01 labels, a celebratory bot comment, and automatic closure. This is a public, optional honor-system check-in—not authentication, certification, or formal grading.

Maintainer verification: Week 1 checkpoint Issue #1 passed and closed automatically. View the successful Action run.

Optional: create and chat with an ADK starter agent

This optional exercise follows the official ADK Python quickstart. It is not part of the setup checkpoint, but it shows what the ADK CLI generates and lets you have one simple terminal conversation.

Terminal · with .venv active
cd "$PROJECTS_DIR/search-agent-lab"
source .venv/bin/activate
adk create my_agent
Model
Choose 1 · gemini-3.5-flash
Backend
Choose 1 · Google AI

Do not choose “Login with Google” for this setup

That option starts a Google Cloud authentication flow and looks for a Cloud or Express Mode project. The learner baseline intentionally uses the simpler Google AI API-key path.

The CLI creates a small agent project:

Expected project structure
my_agent/
├── __init__.py
├── agent.py
├── .env
└── .gitignore

The generated agent has its own .env

ADK stores the selected credential configuration in my_agent/.env. Treat it as a secret: never print, share, or commit it.

Start the generated agent with the official CLI command:

Terminal
adk run my_agent
Example conversation
Running agent root_agent, type exit to exit.
[user]: introduce yourself
[root_agent]: Hello! I am your AI assistant.
[user]: exit

ADK may print experimental credential-service warnings before the prompt. If Running agent root_agent and [user]: appear, the starter agent is ready for a short chat.

Troubleshooting notes

These notes come from real runs of the learner setup.

Google returns 503 UNAVAILABLE or a high-demand message

The model service is temporarily at capacity. Your local environment check is still valid. Wait briefly, then rerun only the live checkpoint cell; do not rebuild .venv or create another API key.
